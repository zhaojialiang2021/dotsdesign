# /shape {想法或需求}

产品设计师的统一入口。无论是"我有个模糊想法"还是"我要在 X 页面加个 Y"，都从这里进。

Skill 会先判断输入属于哪种模式，再选对应流程。你不需要选——直接说出你想做的事。

## 模式判定（Skill 内部，不与用户交互）

读完用户输入后，按这两个特征判定：

| 特征 | 模式 |
|------|------|
| 输入里没有明确的页面/位置/组件，描述偏目标或痛点 | **构思模式**（走 6 问） |
| 输入里有明确的页面名、位置、要加/改的元素 | **描述模式**（直接进方案） |
| 介于两者之间 | 默认走构思模式，但只问最相关的 2-3 个问题 |

不向用户播报模式。直接执行。

## 执行流程

### 1. 读取文档（强制）

按顺序读取：

1. `references/strategy.md` — 确认这个想法属于哪个 Track
2. `references/design-principles.md` — 设计约束 + "为什么"
3. `references/design-tokens.md` — 仅在描述模式后段需要校核引用时读，构思模式可不读
4. 如果是描述模式或已知目标页面：`references/pages/{name}.md` + 该页面引用的组件文档

### 2A. 构思模式：6 问 + 输出页面文档

依次追问以下问题。每个问题用户回答后最多追一次澄清，不纠缠。用户可以跳过或随时说"够了"。

1. **需求现实**：用户现在怎么解决这个？没有 Dots 他们在用什么？
2. **现状痛点**：现有方案的哪个瞬间让人想摔手机？
3. **极端场景**：最迫切需要这个功能的人是什么状态？（等地铁？开会中？刚醒来？）
4. **最窄切口**：这个需求的最小可用形态是什么？砍掉什么还能活？
5. **观察证据**：什么观察让你确信这个需求真实存在？
6. **未来适配**：如果 AI 能力 6 个月后翻倍，这个设计还站得住吗？

回答完后，输出 `references/pages/{name}.md`：

```markdown
---
name: {PageName}
status: draft
last_updated: {YYYY-MM-DD}
used_by: []
track: A | B | C
---

# {页面名称}

## 需求背景
[从问答中提炼，含产品价值假设]

## 用户画像
[最迫切需要的人是谁，处于什么状态]

## 核心场景
[3-5 个具体场景，每个场景一句话]

## 最小形态
[砍到不能砍的版本长什么样]

## AI 交互模型
[这个页面中 AI 扮演什么角色：助手/代理/伙伴/观察者]
[AI 产出与用户内容如何视觉区分]

## 依赖组件

| 组件 | 状态 | 操作 |
|------|------|------|
| ... | ... | ... |

## 开放决策
- [ ] ...
```

### 2B. 描述模式：定位 + 方案 + 组件清单

不进行 6 问，直接进入方案推导。基于设计原则（不是令牌值）输出 PM 可读的方案：

```
## {需求名称}

### 位置
页面 {page} 的 {区域}，紧接在 {已有元素} 之后

### 方案
{一段话描述方案，依据是哪条设计原则}

### 涉及组件

| 组件 | 状态 | 操作 |
|------|------|------|
| Button | complete | 复用，类型 Primary |
| 分享卡片 | 不存在 | 需要新增组件 |

### 关键交互
- 点击 → {行为}
- AI 参与的角色：{助手/代理/伙伴/观察者/不涉及}

### 设计约束
- AI 产出与用户内容是否需要视觉区分
- 品牌色使用场景
- 触觉意图（从 haptics.md 中 5 级里选）
```

### 3. 等待用户决策

- "行" / "继续" → 进入第 4 步建议下一步
- "不对，应该..." → 调整方案，重新输出
- "再问问 X" → 在该问题上展开追问
- "够了" → 终止，不写文档

### 4. 建议下一步

按当前缺口给出**单一最关键**的下一步：

| 当前状态 | 下一步 |
|---------|--------|
| 有新组件未定义 | `/component {name}` 走组件全流程 |
| 组件存在但有 placeholder/draft 的依赖 | `/fill {name}` 补全 |
| 组件全部就绪 | `/preflight {page}` 检查依赖 → `/draw {page}` |
| 方案需要二次审视 | `/review {name}` |

只给一个建议，不列菜单。

## 反模式（不要做）

- 不要在构思模式下急着出方案——6 问是为了暴露真伪，不是走过场
- 不要在描述模式下擅自创造新组件——必须显式提出"需要新组件"让用户决定
- 不要引用具体令牌值（如 `#78AAFA`）——shape 阶段只用规则名，令牌细节属于 `/fill` 和 `/spec`
- 不要把"开放决策"留空——每个新页面都应该有 1-3 个未决项，否则说明思考不充分
