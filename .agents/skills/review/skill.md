# /review {name}

对页面或组件文档做 7 pass 设计审查，每 pass 输出评分 (0-10) 和具体问题。

## 执行流程

### 1. 读取文档（强制）

1. `references/design-tokens.md`
2. `references/design-principles.md`
3. `references/haptics.md`
4. 目标页面文档 `references/pages/{name}.md` 或组件文档 `references/components/{name}.md`
5. 目标引用的已 complete 组件文档

### 2. 视觉验证（如可画板存在）

如果目标有 Paper 画板或 Figma 设计，截图查看当前视觉状态。

### 3. 审查流程

分两层：
- **AI 预检**（自动执行）：二元检查——有没有、对不对、是否引用了存在的令牌
- **人工裁定**（逐项出问题，等待用户判断）：好不好、够不够、审美的边界

AI 做完所有预检后，汇总到一张表里，区分「机器能判的」和「需要你来判的」。人对后者的每项做决定。

---

#### Pass 1：信息架构

AI 预检：
- [ ] 页面是否有清晰的内容分区？（导航区 / 主内容区 / 底部栏）
- [ ] 视觉层级是否定义了标题→正文→辅助文字的完整链路？
- [ ] 用户从哪里进入、可以导航到哪里？（上一页/下一页是否明确）

人工裁定：
- 视觉层级是否反映信息优先级？
- 是否符合 Dots "安静、内容优先" 的调性？

#### Pass 2：交互状态

AI 预检：
- [ ] 三态 Default / Active / Disabled 是否定义？
- [ ] 是否覆盖 Loading / Empty / Error / Success？
- [ ] 触觉意图是否定义？对照 `references/haptics.md` 检查组件的触觉映射
- [ ] Active 态规则：背景降一级（BG-0 → BG-2 或等效）、Label 透明度 0.6、无缩放？

人工裁定：
- 以上检查全部为二元判断。AI 标出不通过项，人确认是否修复。

#### Pass 3：情感弧线

全量人工裁定（AI 不做预检），但 AI 可以辅助提问：
- 用户在这个页面上的情绪轨迹是什么？
- 是否存在"安静的力量"感，还是过度刺激？
- AI 回应出现时，用户的情绪是否有合理过渡？
- 空状态是否给人希望而非沮丧？

备注：情感判断不在 AI 能力范围内，不要在没证据时给评分。不知道就说"不确定"。

#### Pass 4：AI 味检测（核心 pass）

AI 预检（二元检查）：
- [ ] AI 产出与用户输入的视觉区分是否定义？（AI 用 Brand-Blue-Light 背景 + Brand-Blue-Border 边框）
- [ ] AI 生成内容是否标注"由 AI 生成"？
- [ ] AI 回应的 loading 状态是否定义？
- [ ] AI 回应失败时的降级方案是否定义？
- [ ] AI 内容是否可编辑/可删除/可反馈？
- [ ] AI 回应的引用来源是否可追溯？

人工裁定：
- [ ] 是否存在欺骗性 AI 模式？（假装是用户写的、隐瞒 AI 来源、过度拟人化）
- 此项 AI 可以逐条列出嫌疑点，但最终判定靠人。

#### Pass 5：令牌合规

全量 AI 预检：
- [ ] 字号来自令牌表？
- [ ] 颜色引用令牌名（非硬编码 hex）？
- [ ] 圆角来自 5 级体系（6/8/12/20/9999px）？
- [ ] 间距来自 10 级梯度？
- [ ] 卡片只用 BG-0？
- [ ] 品牌色只做点缀（AI 回应气泡除外）？
- [ ] 装饰色只做图标底色？
- [ ] 无装饰性渐变和阴影（Modal 除外）？
- [ ] 分割线默认用 NonOpaque？
- [ ] 交互状态三档？

全部为二元判断，AI 可独立完成。不通过项直接列为「必须修复」。

#### Pass 6：多宽度适配

AI 预检：
- [ ] Mobile (390px)：默认通过
- [ ] 多宽度适配表是否存在？
- [ ] 如有 Tablet/Desktop 变化，是否声明在表中？

不通过项列为「必须修复」。

#### Pass 7：开放决策

全量人工裁定（AI 辅助列出问题）：
- AI 从文档中提取所有标注为「待定」「TODO」「待决策」的项
- 每项 AI 标注：如果推迟，实现者最可能怎么做（最坏情况）
- 标记哪些必须实现前确定，哪些可以后补
- 人对每项做决定
