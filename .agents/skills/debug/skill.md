# /debug {description}

系统性排查问题，4 阶段流程。铁律：无根因不修复。

## 执行流程

### 1. 调查阶段

- 复现问题：按用户描述的操作步骤复现
- 收集信息：控制台错误、网络请求、组件状态
- 截图当前状态
- 确认问题范围：哪个组件/页面受影响

### 2. 分析阶段

- 读取受影响组件/页面的文档（`references/components/` 或 `references/pages/`）
- 对比文档规格与实际实现
- 检查常见 Dots 问题模式：
  - 令牌不匹配（Light/Dark 值混用）
  - 组件状态未正确流转三态模型
  - 触觉反馈未绑定
  - AI 回应/用户输入视觉混淆
  - CSS 变量未定义（检查 `src/index.css`）

### 3. 假设阶段

列出可能的根因，按可能性排序。每个假设附验证方法。

格式：
```
假设 1（最可能）：[描述]
验证方法：[具体操作]
预期结果：[如果假设成立会看到什么]

假设 2：[描述]
验证方法：[具体操作]
预期结果：[...]

...
```

等待用户确认后再进入实施阶段。

### 4. 实施阶段

铁律：**没有确认根因不修复**。

- 选择最可能的假设，执行验证
- 验证通过 → 修复
- 验证失败 → 回到假设 2，继续
- 修复后验证：确认问题已解决且无回归

### 5. 输出排查报告

```
## 排查报告

问题：[描述]
根因：[确认的根因]
修复：[做了什么]
验证：[如何确认修复有效]
预防：[如何避免再次发生]
```
