# /component {name}

全流程：补全组件规格 → 在 Paper 中画展示板 → 用户审阅 → 合规检查 → 归档。

## 执行流程

### 1. 读取文档（强制）

必须重新读取以下文档，不允许凭记忆：

1. `references/design-tokens.md`
2. `references/design-principles.md`
3. `references/haptics.md`
4. 目标组件文档 `references/components/{name}.md`
5. 已 complete 的同级组件文档（用于推导设计语言一致性）

### 2. 评估状态

- **placeholder**：从已有组件的设计语言推导规格，输出草案供用户审阅
- **draft**：检查待补充列表，补全缺口，输出补全方案供审阅
- **complete**：告知用户此组件已完成，询问是否要修订

### 3. 输出规格草案

以表格形式输出规格，包含属性、值、设计备注三列。每条规则附带设计备注解释"为什么"。

等待用户确认：
- "行" → 继续
- "不行，改成..." → 修改后重新输出
- "跳过" → 保留当前状态，终止流程

### 4. 更新组件文档

将确认后的规格写回 `references/components/{name}.md`，更新 status 和 last_updated。

### 5. 画展示板

在 Paper 中创建展示板，结构：

```
标题区：组件名 + 状态标签
───────────────────────
变体段落：每种变体一行
状态段落：Default / Active / Disabled
尺寸段落（如有）
───────────────────────
底部关键规格：一行速查表
```

绘制规则：
- 使用令牌值，禁止硬编码
- 字号、间距、圆角必须来自 design-tokens.md
- 图标必须来自 Bridge 图标库

### 6. 合规检查（强制）

逐条检查：

- [ ] 字号来自令牌表？
- [ ] 颜色引用令牌名而非硬编码 hex？
- [ ] 圆角来自 5 级体系（6/8/12/20/9999px）？
- [ ] 间距来自 10 级梯度？
- [ ] 图标来自指定图标库？
- [ ] 触觉意图已定义？
- [ ] 多宽度适配表已填写？
- [ ] Dark Mode 无需额外规格（令牌层处理）？

任何不通过项必须修复后重新检查。

### 6.5 AI 交互检查（如组件涉及 AI）

如果组件涉及 AI 交互（如消息气泡的 AI 回应模式、Slider 的 AI 人格调节），额外检查：

- [ ] AI 产出与用户输入的视觉区分是否定义？
- [ ] AI 回应的 loading 状态是否定义？
- [ ] AI 生成内容是否需要标注？
- [ ] 在组件文档中增加"AI 交互说明"段落

### 7. 归档

用 `/archive` 将 Paper 展示板的代码写回组件文档的 `## Paper 代码` 段落，status 改为 `complete`。
