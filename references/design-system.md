# 点点（Dots）设计规范

> 本规则文件定义了小红书点点 AI 助手的设计规范。
> 所有 vibe coding 产出的页面/组件必须遵循以下规范。
> 颜色体系以 Figma Dots-Tokens 为准，排版体系以富文本渲染规范为准。

---

## 0. Harness Engineering 底层逻辑

Dots 设计系统的底层逻辑是 Harness Engineering：给 AI 搭一套可执行、可验证、可复用的工程轨道。传统组件库只回答“组件长什么样”；Dots 组件库还要回答“AI 什么时候用它、能传什么参数、哪些状态必须覆盖、生成后怎么检查”。

### 0.1 五层结构

| 层级 | 作用 | 产物 |
|------|------|------|
| Design Tokens | 定义视觉物理规则 | `tokens/*.json`、`tokens.css` |
| Component Harness | 定义单个组件的语义、结构、状态、行为、验证 | `components/*.schema.json`、组件文档、实时预览 |
| Pattern Harness | 定义多个组件如何组成业务模式 | 对话流、Skill 选择、AI 卡片生成 |
| Page Harness | 用真实页面验证组件和模式是否能跑通 | `dotted-demo` 等页面 demo |
| Agent Harness | 告诉 AI 如何读取、生成、修改、检查 | `/skill.md`、`components.json`、Copy for AI |

### 0.2 Component Harness 的完成标准

每个核心组件必须包含：

- **视觉规范**：颜色、字体、圆角、间距、阴影和状态。
- **语义定义**：这个组件解决什么问题，什么时候该用，什么时候不该用。
- **结构 Schema**：AI 可用的 props、slot、variant、状态枚举。
- **行为规则**：点击、展开、关闭、滚动、发送、加载、禁用等边界。
- **示例场景**：组件在真实页面里的用法，而不是孤立样式。
- **验证方式**：lint、build、截图检查、设计 checklist、Figma 对齐。
- **Agent 使用说明**：AI 生成时必须怎么用，不能怎么改，失败后怎么修。

### 0.3 判断标准

一个组件只有 CSS，不算进入系统；只有 Figma，不算进入系统；只有页面 demo，也不算进入系统。只有当它能被人读、被 AI 生成、被代码复用、被检查验证，才算完成。

## 3. 字体体系

### 3.1 字体栈
组件规范、Dots App UI、页面 demo 统一使用 PingFang SC。文档站首页、导航和说明文字可以使用展示字体，但不能影响组件预览内部。

```css
font-family: 'PingFang SC', -apple-system, BlinkMacSystemFont, 'SF Pro Display', 'Helvetica Neue', sans-serif;
```

Docs / Studio / Landing 等非规范展示 UI 使用独立字体边界：

```css
--font-display-cn: 'GlowSans SC', 'PingFang SC', -apple-system, BlinkMacSystemFont, sans-serif;
--font-display-en: 'PhonicTrial', 'GlowSans SC', 'PingFang SC', sans-serif;
--font-mono: 'SaansMono-TRIAL', 'SFMono-Regular', Consolas, monospace;
```

规则：组件页的真实组件、Dotted 页面 demo、MessageBubble、Button、Input、LiveWaveform、Sheet 必须使用 PingFang SC。看到「组件」或「页面 demo」就回到组件字体栈，不跟随文档展示字体。

## 6. 动画规范

### 6.1 缓动曲线
```css

```

### 6.2 时长

| 场景 | 时长 |
|------|------|
| 元素入场 | 0.25s |
| 内容展开 | 0.35s |
| 延迟入场 | 0.3s |
| 元素退场 | 0.25s |
| 弹窗出入 | 0.35s～0.45s |

### 6.3 标准入场动画
```css
@keyframes fadeUp {
  from { opacity: 0; transform: translateY(20px); }

```

### 6.4 交互反馈

- **按压缩放**：`transform: scale(.95)` ~ `scale(.97)`
- **按压透明**：`opacity: .5`
- **禁用选中**：`-webkit-user-select: none`
- **禁用高亮**：`-webkit-tap-highlight-color: transparent`

## 8. 富文本内容规则

> 来源：点点 AI 富文本渲染规范
> 核心理念：模型是排版决策者，Markdown 是序列化格式，内容类型 + 层级规则 = 完整组合系统。

### 8.1 层级结构

```
总标题（H1，可选，全文唯一）
└── 一级模块（H2，可多个）
      └── 二级模块（H3，可多个）
            └── 三级模块（H4，可多个）
                  └── 原子内容
```

**标题映射：**

| 业务语言 | 技术语言 | 字号 |
|---------|---------|------|
| 总标题 | H1 `#` | 20pt |
| 一级模块标题 | H2 `##` | 18pt |
| 二级模块标题 | H3 `###` | 17pt |
| 三级模块标题 | H4 `####` | 16pt |

- 原子只能放在三级模块里
- 一级、二级模块是容器，只能包含子模块
- 模块标题是可选的，技术上可有可无
- 模块边界通过显式标记区分，标记格式由研发定义

### 8.2 内容类型

**文字类**（信息主要通过文字传递）

| 类型 | Markdown 符号 |
|------|--------------|
| 标题 | `#` `##` `###` `####` |
| 段落 | 纯文本 |
| 无序列表 | `- item` |
| 有序列表 | `1. item` |
| 任务列表 | `- [ ] item` |
| 引用块 | `>` |

**内容块类 — 纯展示型**（固定格式，内容不由模型控制，附带跳转能力）

| 类型 | 来源 |
|------|------|
| 图片 | 业务自定义 |
| 图组 | 业务自定义 |
| 地图 | 业务自定义 |
| 商品卡 | 业务自定义 |
| 地点卡 | 业务自定义 |
| 社区用户声音 | 业务自定义 |
| 表格 | Markdown 原生 |

**内容块类 — 容器型**（内部可自由组合文字类原子，不能再嵌套内容块）

| 类型 | 说明 |
|------|------|
| 高亮块 | 可放：标题、段落、列表、引用块。用于强调重点内容 |

**分隔元素**

| 类型 | 符号 |
|------|------|
| 分割线 | `---` |

**行内元素**（可叠加在：段落、列表项、引用块、高亮块内的文字、表格单元格）

| 类型 | 符号 | 性质 |
|------|------|------|
| 粗体 | `**text**` | 样式类 |
| 斜体 | `*text*` | 样式类 |
| 粗斜体 | `***text***` | 样式类 |

| 行内实体标签 | 业务自定义 | 交互类，点击跳转实体详情页 |

### 8.3 硬规则

**结构与层级**
- 模块最多三层，三级模块下直接是原子，不能再有子模块
- 总标题全文唯一，不与一级模块标题混用
- 富文本允许完全没有结构，纯段落展示也可以

**内容块排列**
- 两个内容块之间必须有文字，内容块不能相连
- 内容块可以作为模块的第一个或最后一个元素
- 内容卡不可出现在列表内、高亮块内
- 可罗列内容卡（地点卡、商品卡、社区用户声音等）可以并排累加
- 不可罗列内容卡（地图等大型内容卡）不可并排

**列表规则**
- 列表不支持嵌套
- 列表项内不能放内容块
- 列表项内不能有多个段落

**引用块规则**
- 引用块内不能嵌套其他文字类原子，只能叠加行内元素
- 引用块内不能有标题

**高亮块规则**
- 高亮块内只能放文字类原子
- 高亮块不能嵌套高亮块
- 高亮块不能连续出现

**分割线 / 表格规则**
- 分割线不能连续出现
- 表格单元格内只能放文字和行内元素

### 8.4 写作原则（模型应内化的排版素养）

- 内容块出现前后需有文字引导或承接，不能突兀插入
- 文章开始时建议先有引导性文字，再展开结构
- 标题服务于区分，不服务于装饰；只有需要与同级内容区分时才加标题
- 结构服务于表达，不替代表达；不要用结构树代替叙述
- 以模块为单位思考，不以原子为单位

### 8.5 富文本左右间距

| 类型 | 左右间距 |
|------|---------|
| 文字类原子 | 各 24pt |
| 大内容块（图片、图组、地图、商品卡、地点卡等） | 各 14pt |
| 高亮块 | 各 14pt（内部 padding 待定） |
| 引用块 | 左缩进 14pt |

## 12. 设计原子组合规则

> 当你需要创造一个全新的、规范里没有现成模板的组件时，遵循以下规则。
> 这些规则确保新组件在视觉上"属于点点"。

### 12.1 背景层级规则

背景色从底层到顶层逐级提亮，形成视觉纵深：

```
Bg (最底层，页面)
 └─ Bg 1 (中间层，栏/工具条)
     └─ Bg 3 / Fill 5 (顶层，卡片/面板/弹窗)
         └─ Inverted Fill 1~2 (卡片内的子区域)
```

**规则**：容器的背景永远比它的父容器亮一级。不要在 `Bg` 上直接放 `Inverted Fill` 内容。

### 12.2 文字层级规则

文字不靠换色区分层次，靠同一基色 + 不同 opacity：

```
Title (100%)     → 标题、正文、关键信息
Paragraph (80%)  → 副标题、段落、图标
Description (60%) → 描述、辅助信息、标签
Description Lighter (40%) → 更轻的提示
Placeholder (16%) → 占位、loading
Disabled (9%)    → 禁用态
```

**规则**：同一个容器内最多使用 3 级文字层次（Title + Description + Placeholder）。不要在一个小组件里用满 6 级。

### 12.3 圆角递减规则

圆角从外层到内层递减，形成包含关系：

```
设备外壳: 54px
 └─ 富文本卡片: 36px
     └─ 气泡/普通卡片: 22px
         └─ 输入框/标签: 16px / 12px
             └─ 内部元素(图片/按钮): 8px
```

**规则**：内层圆角不能大于外层。如果新建一个卡片（22px），它内部的元素最大用 14px 或 8px。

### 12.4 间距节奏规则

间距遵循"递减三拍"的节奏：

```
一级模块间距: 32pt  → 用于顶级并列模块之间
二级模块间距: 22pt  → 用于模块内部的子模块之间
三级模块间距: 16pt  → 用于最细粒度的元素之间
```

**补充间距值：**
- 元素紧凑排列：8px（如标签之间、用户连续气泡）
- 段落内行距：1.7 倍行高
- 容器内边距：上 28pt，左右 24pt（文字）/ 14pt（大内容块）

**规则**：不要使用 32/22/16/8 之外的间距值。对话流中不同侧消息用 20px，同侧连续消息用 8~16px。

### 12.5 阴影克制规则

```
shadow 1 (4%)   → 卡片、浮层容器（大部分场景用这个）
shadow 2 (12%)  → 面板、重要浮层
shadow 3 (20%)  → 极少使用，仅全屏面板
```

**规则**：90% 的场景用 `shadow 1`。不要自定义阴影值。暗色模式下阴影 opacity 翻倍（4%→8%, 12%→24%）。

### 12.6 边框使用规则

```
Separator (75% 白色)  → 用户气泡边框（亮色模式下几乎不可见）
Separator 2 (8%)      → 最常用：AI 气泡、卡片内部元素、分割线
Separator 3 (10%)     → 输入框边框
Separator 4 (20%)     → 较重的容器边框
```

**规则**：边框粗细统一 0.5px。不要使用 1px 以上的边框（按钮除外）。当有阴影时，边框可以省略。

### 12.7 新建卡片的标准配方

当你需要创建一个新的卡片组件时，按以下配方：

```css
.new-card {
  background: var(--fill5);          /* Fill 5 */
  border-radius: 22px;               /* 卡片级圆角 */
  box-shadow: var(--shadow-card);    /* shadow 1 */
  overflow: hidden;
}

.new-card__header {
  padding: 16px 20px 12px;
  color: var(--description);         /* Description */
  font-size: 13px;
}

.new-card__title {
  padding: 0 20px;
  color: var(--title);               /* Title */
  font-size: 16px;
  font-weight: 600;
}

.new-card__content {
  padding: 12px 14px;               /* 大内容块左右 14px */
}

  padding: 12px 20px 16px;

}

  border-radius: 8px;               /* 内部元素圆角 */
```

## 13. 暗色模式适配指南

> 所有新组件必须同时支持暗色模式。以下是快速适配方法。

### 13.1 核心原则

暗色模式不是简单取反。遵循"基色翻转 + 降低对比度"原则：

| 亮色 | 暗色 | 说明 |
|------|------|------|
| `#141414` 基色 | `#FFFFFF` 基色 | 文字基色翻转 |
| `#FFFFFF` 背景 | `#323232` 背景 | 背景用深灰，不用纯黑 |
| `#F3F3F3` 页面底 | `#242424` 页面底 | 底层背景变暗 |
| opacity 保持不变 | opacity 保持不变 | 透明度体系不变 |

### 13.2 CSS 变量适配模板

```css
@media (prefers-color-scheme: dark) {
  :root {
    --bg: #242424;
    --bg1: #282828;
    --bg2: #2D2D2D;
    --bg3: #323232;
    --fill5: rgba(50,50,50,.8);
    --fillA: #323232;
    --fillB: #1C1C1C;
    --title: rgba(255,255,255,.94);
    --paragraph: rgba(255,255,255,.8);
    --description: rgba(255,255,255,.6);
    --description-lighter: rgba(255,255,255,.4);
    --placeholder: rgba(255,255,255,.16);
    --separator2: rgba(255,255,255,.08);
    --shadow-card: 0 4px 10px rgba(20,20,20,.08);
  }
}
```

## 14. 最佳实践

1. **移动优先**：所有 demo 默认面向 iOS 设备尺寸（430×932）
2. **使用 Token**：所有颜色使用 Dots-Tokens 定义的 token，不硬编码色值
3. **支持暗色**：新组件需同时提供 Light/Dark Mode 适配
4. **间距节奏**：模块间距遵循 32→22→16 的递减节奏
5. **阴影克制**：使用 shadow 1/2/3 三级阴影，不自定义
6. **圆角层次**：外层大圆角（36px），内层小圆角（14px），形成包含关系
7. **文字层次**：通过 opacity 控制层次（100%→80%→60%→40%→16%→9%）
8. **触摸友好**：按钮最小 44px 触摸区域，提供按压缩放反馈
9. **先查模板**：构建已知场景时，先查第 11 节的映射表
10. **再用规则**：构建未知场景时，遵循第 12 节的组合规则
11. **配方优先**：新建卡片/列表用第 12.7/12.8 的标准配方起步
