# Notifications 系统通知

Dots 的通知不是营销推送，是异步对话的自然闭环。用户发起一个 AI 任务，AI 完成后告知一声——仅此而已。

---

## 推送场景白名单

Dots 只有两种场景可以推送系统通知。白名单以外的任何场景，不管产品策略如何演进，都不推。

| # | 场景 | 触发条件 | 优先级 |
|---|------|---------|--------|
| 1 | AI 异步回应完成 | 用户主动发起的 AI 任务处理完毕 | 默认 |
| 2 | 用户定时提醒 | 用户对某条记忆设置的提醒时间到达 | 高 |

**明确禁止的场景**：每日回顾、功能推荐、连续未使用提醒、记忆留存率下降、系统自动生成的总结、任何用户未主动发起的推送。

设计备注：用户只能被他自己的动作触发通知——要么他要求 AI 做了事，要么他设置了提醒。系统没有资格发起对话。

---

## 通知内容规则

### AI 回应完成

| 属性 | 值 |
|------|-----|
| 标题 | 「你的一个问题有回应了」 |
| 正文 | 不超过 20 个字，是线索不是摘要 |
| 正文示例 | 「会议记录整理好了」 |
| 正文反例 | 「你上周三和张三讨论的 Q2 预算会议记录已整理完毕，包含…」 |
| 按钮 | 仅一个：「打开看看」 |
| 标题风格 | 无感叹号、无 emoji、不重复品牌名 |

正文只说类型不说内容。用户知道他问过什么，不需要在通知里复述一遍。

### 定时提醒

| 属性 | 值 |
|------|-----|
| 标题 | 用户设定的提醒标题 |
| 正文 | 关联记忆的引用片段，不超过 30 字 |
| 按钮 | 「查看」 |

### 通知按钮通用规则

通知上只提供一个正向操作按钮。不提供「标记已读」「稍后提醒」「删除」等辅助操作——通知不是小界面，是一扇门。

---

## 推送控制参数

### 频率限制

| 参数 | 值 |
|------|-----|
| 每日最大推送数（不含定时提醒） | 3 条 |
| 超过上限行为 | 丢弃，不合并 |
| 定时提醒不受限 | 是 |

设计备注：3 条是上限不是目标。如果用户一天收到 3 条推送，说明他今天频繁在和 AI 互动——这种情况他大概率已经自己在 app 里看到了，实际推送数自然会低于上限。超过 3 条直接丢弃而不是合并，是因为合并推送的语义是「你有几件事没看」——这本身就是一种压迫。

### 聚合窗口

| 参数 | 值 |
|------|-----|
| 窗口时长 | 15 分钟 |
| 窗口内多个 AI 回应完成 | 合并为 1 条，标题改为「你有 N 个问题有回应了」 |
| 窗口内用户主动打开 app | 窗口清零，所有待推送丢弃 |

### 静默时段

| 参数 | 值 |
|------|-----|
| 默认起始 | 22:00 |
| 默认结束 | 08:00 |
| 用户可自定义 | 是（在 AI 偏好页中） |
| 静默期间行为 | 所有推送延迟到静默结束后，期间完成的 AI 回应只保留最新一条 |
| 定时提醒是否静默 | 不受静默时段限制，准时触发 |

## 触觉反馈

| 通知类型 | 意图 | 强度 | 持续 |
|---------|------|------|------|
| AI 回应完成 | tick | 0.3 | 10ms |
| 定时提醒 | light | 0.5 | 15ms |

设计备注：AI 回应比定时提醒更轻。定时提醒是用户在特定时间明确要求的打断——这是被授权的打断。AI 回应是不指定时间的，tick 的语义是「有东西但你不必理会」——更轻才是更尊重。

## 通知之外：不说话的感知

Dots 不依赖系统通知作为唯一的信息传达通道。以下方式优先级高于推送：

### Live Activity / 灵动岛

AI 处理中显示「AI 正在整理…」，完成后自动更新为「整理好了」。用户没有被通知，他就是知道——在岛上看一眼就是全部。

| 属性 | 值 |
|------|-----|
| 处理中文案 | 「AI 正在整理…」 |
| 完成文案 | 「整理好了」 |
| 是否需要推送配合 | 否。Live Activity 完成后不触发系统通知 |
| 适用条件 | AI 任务提交后用户离开 app |

### 主屏幕小组件

| 属性 | 值 |
|------|-----|
| 显示内容 | 最近一条待查看的记忆片段（一行文字） |
| 不显示 | 红点数字、badge count |
| 点击行为 | 进入 app，打开对应记忆 |

设计备注：数字是压迫，片段是邀请。5 条待看记忆用红点表示是焦虑，用一句话表示是期待。

### 应用内提醒（非推送）

| 类型 | 位置 | 自动消失 | 触觉 |
|------|------|---------|------|
| AI 回应完成 | 首页消息流自然出现，不额外通知 | — | — |
| 每日回顾可看 | 首页顶部一条安静提示条 | 否，用户主动关闭 | tick |
| 新功能引导 | 相关功能入口旁一个蓝点 | 是，点击后就消失 | — |

## 交互流程

```
用户提交 AI 任务
       │
       ▼
┌──────────────┐
│ AI 开始处理   │ → Live Activity: 「AI 正在整理…」
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ AI 处理完毕   │
└──────┬───────┘
       │
       ├─ 用户在 app 内？ ──── 是 ──→ 消息流自然出现，不推
       │
       └─ 用户在 app 外？
              │
              ├─ 静默时段？ ──── 是 ──→ 暂存，时段结束后推（只保留最新一条）
              │
              └─ 非静默时段
                     │
                     ├─ 聚合窗口内有其他待推送？
                     │        │
                     │        └─ 是 ──→ 等待窗口结束，合并为一条
                     │
                     └─ 否 ──→ 等待 15 分钟窗口
                                   │
                                   ├─ 期间用户打开 app ──→ 丢弃
                                   │
                                   └─ 窗口结束 ──→ 推送
```

## 与现有设置项关系

侧边栏的「通知」Switch 开关控制的是：是否允许系统推送。在侧边栏关闭通知代表着所有推送（包括定时提醒）都不发。需要提醒用户「关闭通知后，AI 回应的完成提醒和定时提醒都不会推送」——这是一个强语义的开关off，不是静音。

未来 AI 偏好页可以增加细粒度控制（仅定时提醒、仅 AI 回应、自定义静默时段），但 MVP 阶段侧边栏一个总开关就够了。

## 多宽度适配

| 断点 | 变化 | 说明 |
|------|------|------|
| Mobile | 无变化 | 标准推送 + Live Activity |
| Tablet | 无变化 | 推送行为一致，小组件尺寸适配 |
| Desktop | 无变化 | macOS 通知中心 |

通知行为与设备无关。Live Activity 和小组件仅移动端支持。
