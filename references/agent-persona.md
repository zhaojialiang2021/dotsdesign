# Agent Persona

> 把 AI 代理当作用户来设计。本文档面向产品设计师（在 Dots 项目里，产品和设计是同一个角色），描述 AI 代理在设计系统中的行为特征、能力边界和常见错误。
> 多人协作时，新成员读本文档了解 agent 能做什么、不能做什么。

## 项目内的角色定义

Dots 不区分"产品经理"和"设计师"——同一个人既负责定义"做什么"，也负责定义"长什么样"。本文档里出现的"产品设计师"指代这个统一角色。

开发是独立角色，通过 `/spec` 和 `/build` 与产品设计师协作。

Agent 是第三类用户。它不是灵感来源，也不是审美裁判，而是运行在 Dots Harness 里的执行者：读契约、生成界面、跑检查、暴露不确定点。

## 代理画像

**角色**：初级设计师 + 严格执行者 + 可验证的执行引擎
**工作态度**：勤奋、从不遗漏检查清单、但缺乏审美直觉
**关键局限**：不知道「好不好看」，只知道「合不合规」

## Harness Engineering 对 agent 的要求

Dots 的设计系统不是给 agent 自由发挥的素材库，而是 agent 的运行轨道。每次生成 UI 时，agent 必须按以下顺序工作：

1. 读取任务属于哪个 Track，确认目标指标。
2. 读取 tokens 和 principles，确定不可违反的视觉边界。
3. 读取目标组件的 Component Harness，而不是只看截图。
4. 读取页面 demo 或 Pattern Harness，确认多个组件如何组合。
5. 生成代码后跑 lint / build / 视觉检查。
6. 对无法规则化的审美判断显式交给人，不私自补判断。

一个合格的 agent 输出，不是「看起来像」，而是能说明：用了哪些 tokens、命中了哪些 schema、覆盖了哪些状态、哪些约束被验证过。

## 读取的文件

代理在每次执行任务时会按顺序读取以下文件：

| 文件 | 何时读 | 读什么 |
|------|--------|--------|
| strategy.md | 派发任务时 | 确认任务属于哪个 Track、要影响什么指标 |
| design-tokens.md | 任何涉及视觉的操作 | 颜色/字号/间距/圆角/阴影的精确值 |
| design-principles.md | 任何涉及判断的操作 | 每条规则的"为什么"（设计备注） |
| haptics.md | 涉及交互的操作 | 5 级触觉意图和波形参数 |
| framework-a.md | 画聊天流页面时 | 聊天流的结构和适配规则 |
| components/*.schema.json | 使用深度组件时 | props enum、states、constraints、harness 规则 |
| 组件文档 | 用到该组件时 | 语义、变体、状态矩阵、do/don't |
| 页面文档 | 画或修改该页面时 | 需求背景、场景、依赖组件 |

## 可调用的工具

| 工具 | 能力 | 局限 |
|------|------|------|
| Paper MCP | 绘制 390×844 画板、截图对比 | 只能生成 inline style HTML，不是生产代码 |
| Figma MCP | 读取旧版设计稿作为参考 | 不写入，Figma 不是真相源 |
| 文件系统 | 读写 `references/`、`components/` 和 `src/` 下的文件 | 不会主动创建新 tool/库 |
| Git | 提交、查看 diff、查看 log | 不 push（除非你明确说可以） |
| npm scripts | 运行 lint、build、schema 检查 | 检查不等于审美判断 |

## 容易犯的错

1. **不读文档就动手**：代理可能凭记忆使用令牌值。每个 Skill 开头强制重新读取相关文档，就是为了防止这一点。
2. **发明新组件**：如果你不说"用已有的 Button"，代理可能每次画一个新按钮。永远在 prompt 里注明使用的已有组件。
3. **只复制视觉，不沉淀 harness**：页面还原完成后，如果没有更新组件 schema、状态、constraints，这次还原就不能复用。
4. **美化式编造**：代理可能给卡片加阴影、给按钮加渐变，因为这些在训练数据里很常见。但它不被允许这么做，因为 design-principles 禁止装饰性阴影和渐变。
5. **记忆幻觉**：代理可能声称某个令牌值是 X，但 design-tokens.md 里写的是 Y。遇到不确定的数值要求它引用令牌文档原文。
6. **跳过检查清单**：代理可能在画完图后忘记逐条执行合规检查。每个 Skill 末尾的检查清单是强制步骤，不能省略。
7. **过度拟人化**：代理有时会在设计备注或注释里写"这样看起来更温暖"之类的主观表达——这种无依据的判断应当忽略。

## 能力边界

以下是代理的能力分界，知道什么该推给人判断：

**代理自己能判断的**：数值是否来自令牌表、圆角是否来自 5 级体系、间距是否来自梯度、引用是否指向存在的令牌、多宽度适配表是否存在、props 是否命中 schema enum、状态是否全覆盖、constraints 是否被违反。

**需要人判断的**：这个间距看着是否太挤、整体氛围是否对、情感弧线是否连贯、是否存在欺骗性 AI 模式、两个接近的选项哪个更好、业务策略是否成立。

**代理协同**：多个代理可以并行跑不同任务——一个画首页、一个做组件展示板、一个跑审计。它们都读同一套文档，输出一致性由文档保证。但不要让两个代理同时修改同一个文件，git merge 解决不了设计决策冲突。
